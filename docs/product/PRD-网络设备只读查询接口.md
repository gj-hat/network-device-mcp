## 一、背景与目标

### 1.1 背景

运维和技术人员需要频繁登录网络设备（交换机、路由器）查看配置和状态信息。当前操作方式存在以下问题：

- 需要逐台 SSH 登录，效率低
- 人工操作有误操作风险，可能执行到配置变更命令
- 缺乏统一的查询入口

### 1.2 产品目标

提供一个 MCP（Model Context Protocol）标准接口，让 AI 助手（如 Claude Code）能够代替人工安全地查询网络设备信息，且**绝对不会执行任何配置变更操作**。

### 1.3 核心原则

**安全第一**：本系统唯一且最高优先级是确保只读。宁可拒绝合法查询，也不能放过一条危险命令。

## 二、用户与场景

### 2.1 目标用户

- 运维工程师：通过 AI 助手批量查询设备状态
- 网络工程师：快速获取多台设备的配置信息做对比分析
- 技术管理者：了解设备运行概况

### 2.2 典型使用场景

| 场景 | 用户说 | 实际发生 |
|------|--------|----------|
| 查看设备版本 | "帮我看一下这台交换机的版本" | AI 从命令菜单选择 `show_version` → 调用 MCP 执行 → AI 整理结果告诉用户 |
| 查找 UP 接口 | "哪些接口是 UP 的？" | AI 选择 `show_interfaces_brief` → 执行 → AI 分析输出筛选出 UP 的接口 |
| 批量巡检 | "帮我看看这 10 台设备的 CPU 使用率" | AI 选择对应命令 → 批量执行 → AI 汇总分析 |
| 排查故障 | "从这台设备 ping 一下 10.0.0.1" | AI 选择 `ping_test`，填入参数 target=10.0.0.1 → 执行 → AI 解读结果 |
| 对比配置 | "帮我对比两台设备的路由表" | AI 分别选择 `show_route` 执行 → AI 对比分析差异 |
| 设备巡检 | "帮我全面巡检一下这台设备" | AI 选择多个命令（版本、CPU、内存、接口）→ 一次连接执行多条 → AI 汇总分析 |

### 2.3 系统职责分工

| 角色 | 职责 |
|------|------|
| **用户** | 用自然语言描述需求 |
| **AI 助手（Claude）** | 理解用户意图 → 从命令菜单选择 command_id → 填写参数 → 调用 MCP 工具执行 → 分析原始结果 → 用自然语言回答用户 |
| **MCP 服务端（本系统）** | 提供命令菜单 → 安全校验 → 拼装命令 → SSH 执行 → 返回原始文本 → 记录日志 |

> **关键设计**：服务端不做自然语言理解，也不做结果分析。服务端保持轻量，只负责**安全地执行命令并返回原始文本**。智能分析由 AI 助手完成。
>
> **命令配置化**：AI 不能自由输入命令字符串，只能从预定义的命令菜单中选择 `command_id` 并填写参数。具体命令由服务端从配置文件读取并拼装，AI 不参与命令构造。这从根本上消除了 AI 构造危险命令的可能性。

## 三、功能需求

### 3.1 功能一：单设备查询

**描述**：对指定网络设备执行一条只读命令，返回原始执行结果。

**输入参数**：

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| host | 是 | 设备 IP 地址 |
| device_type | 否 | 设备类型：cisco / huawei / h3c，默认 cisco |
| command_id | 是 | 命令标识（从命令菜单获取） |
| params | 否 | 命令参数，key-value 形式 |
| port | 否 | SSH 端口，默认 22 |
| username | 否 | SSH 用户名，可覆盖服务端默认凭据 |
| password | 否 | SSH 密码，可覆盖服务端默认凭据 |

**输出**：

```json
{
  "host": "192.168.1.1",
  "success": true,
  "command_executed": "show version",
  "output": "命令原始输出文本",
  "error": ""
}
```

**关键规则**：
- `command_id` 必须在命令配置中存在，不存在的直接拒绝
- 参数经过类型校验和注入防护后，由服务端拼装最终命令
- AI 不直接接触命令字符串
- 输出为设备返回的原始文本，不做任何解析或加工

### 3.2 功能二：多设备批量查询

**描述**：对多台设备并发执行同一条只读命令，汇总返回各设备的结果。

**输入参数**：

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| hosts | 是 | 设备 IP 地址列表 |
| device_type | 否 | 设备类型，默认 cisco |
| command_id | 是 | 命令标识 |
| params | 否 | 命令参数 |
| port | 否 | SSH 端口，默认 22 |
| username | 否 | SSH 用户名，可覆盖服务端默认凭据 |
| password | 否 | SSH 密码，可覆盖服务端默认凭据 |

**输出**：每台设备的执行结果列表，格式同 3.1。

**关键规则**：
- 命令校验只需做一次（相同 command_id + params），校验失败则所有设备都不执行
- 各设备并发执行，单台失败不影响其他设备
- **并发上限：最多同时连接 50 台设备**，超出部分排队等待

### 3.3 功能三：查看命令菜单

**描述**：查询指定设备类型可用的命令列表。AI 助手通过此工具了解"菜单"，从而选择合适的命令。

**输入参数**：

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| device_type | 否 | 设备类型，默认 cisco |

**输出**：命令列表，包含 command_id、名称、描述、参数定义。

```json
[
  {
    "command_id": "show_version",
    "name": "查看设备版本",
    "description": "显示设备软件版本、硬件型号、运行时间等信息",
    "params": []
  },
  {
    "command_id": "ping_test",
    "name": "Ping 测试",
    "description": "从设备发起 ping 测试",
    "params": [
      { "name": "target", "type": "ip_address", "required": true, "description": "目标 IP 地址" }
    ]
  }
]
```

**说明**：
- 命令列表在服务端配置文件（`config/commands.yaml`）中定义
- 扩展命令只需编辑配置文件，不需要修改代码
- AI 助手应在需要时先调用此工具获取可用命令，再选择合适的 command_id

### 3.4 功能四：多命令单连接执行

**描述**：对一台或多台设备，在一次 SSH 连接中执行多条只读命令，减少重复建立连接的开销。适用于巡检等需要同时获取多项信息的场景。

**输入参数**：

| 参数 | 是否必填 | 说明 |
|------|----------|------|
| host / hosts | 是 | 单台设备 IP 或设备 IP 列表 |
| device_type | 否 | 设备类型，默认 cisco |
| commands | 是 | 命令列表，每项包含 command_id 和可选 params |
| port | 否 | SSH 端口，默认 22 |
| username | 否 | SSH 用户名，可覆盖服务端默认凭据 |
| password | 否 | SSH 密码，可覆盖服务端默认凭据 |

**commands 格式示例**：

```json
[
  { "command_id": "show_version", "params": {} },
  { "command_id": "show_interfaces_brief", "params": {} },
  { "command_id": "ping_test", "params": { "target": "10.0.0.1" } }
]
```

**输出**：按命令逐条返回结果，AI 能识别哪个结果对应哪条命令。

```json
{
  "host": "192.168.1.1",
  "results": [
    {
      "command_id": "show_version",
      "command_executed": "show version",
      "success": true,
      "output": "...",
      "error": ""
    },
    {
      "command_id": "show_interfaces_brief",
      "command_executed": "show ip interface brief",
      "success": true,
      "output": "...",
      "error": ""
    },
    {
      "command_id": "ping_test",
      "command_executed": "ping 10.0.0.1",
      "success": true,
      "output": "...",
      "error": ""
    }
  ]
}
```

**关键规则**：
- 每条命令独立进行安全校验，某条校验失败不影响其他命令执行
- 一次 SSH 连接中按顺序执行所有通过校验的命令
- 校验未通过的命令在结果中标记 `success: false` 并说明拒绝原因
- 批量设备（hosts 列表）时，每台设备各自建立一次连接执行所有命令，设备间并发

## 四、安全需求（最高优先级）

### 4.1 封闭集合安全模型

采用**命令配置化**方案，安全性从根本上优于开放式过滤：

- AI **不能**自由输入命令字符串，只能从配置文件的封闭集合中选择 `command_id`
- 命令的主体部分锁死在配置中不可变，AI 只能补充参数值
- 命令拼装由服务端完成，AI 不参与

### 4.2 安全校验流程

校验在命令发送到设备**之前**完成：

1. **command_id 校验** — 不在配置中的 command_id 直接拒绝
2. **参数校验** — 必填检查、类型校验（ip_address / string / integer）、范围校验
3. **参数注入防护** — 参数值不得包含 `;` `|` `&` `` ` `` `$` `\n` `\r` `\\` 等特殊字符
4. **服务端拼装命令** — 主命令 + 校验后的参数值
5. **兜底黑名单** — 拼装后的最终命令仍检查危险关键词（`config`, `delete`, `write`, `reboot` 等），作为防御纵深

> 理论上第 5 步不会触发（因为主命令是预定义的安全命令），如果触发说明配置有问题。

### 4.3 凭据安全

- 服务端 `.env` 中维护一组默认凭据：`SSH_USERNAME` / `SSH_PASSWORD`
- 客户端可通过 `username` + `password` 参数覆盖默认凭据
- 优先级：客户端传入 > 服务端默认值
- `.env` 缺少默认凭据且客户端未传入时，返回明确错误
- `.env` 文件不得纳入版本控制
- 仅支持密码认证

### 4.4 安全原则

- **默认拒绝**：任何不能确认安全的命令，宁可拒绝也不执行
- **校验前置**：所有安全校验在 SSH 连接建立之前完成
- **不可绕过**：调用方没有任何方式跳过安全校验
- **封闭集合**：命令来源是配置文件的有限集合，而非开放式输入

## 五、非功能需求

### 5.1 技术约束

| 项目 | 要求 |
|------|------|
| 开发语言 | Python |
| 接口协议 | MCP（Model Context Protocol） |
| 传输模式 | SSE（Server-Sent Events），远程部署 |
| 连接方式 | SSH（仅密码认证） |
| 并发支持 | 异步并发，最大 50 连接 |

### 5.2 部署模式

- MCP 服务部署在**远程服务器**上（能访问网络设备的机器）
- 客户端（AI 助手）通过 URL 远程连接 MCP 服务
- 首期暂不做接入认证，直接 HTTP + SSE 通信
- 生产环境建议启用 HTTPS

### 5.3 性能要求

| 指标 | 目标 |
|------|------|
| 单设备命令响应 | 30 秒内超时 |
| 批量执行 | 各设备并发执行，不串行等待 |
| 并发上限 | 50 台设备同时连接 |

### 5.4 日志要求

- **日志位置**：项目目录下 `logs/audit.log`
- **日志内容**：每次操作记录以下信息：
  - 时间戳
  - 目标设备（host:port）
  - 设备类型
  - 执行的 command_id 和参数
  - 最终拼装的命令
  - 凭据来源（default / client）
  - 执行结果（成功/失败）
  - 是否被安全校验拦截，拦截原因
- **日志格式**：每行一条 JSON 记录，便于后续分析
- **日志轮转**：暂不要求，后续根据日志量决定

### 5.5 支持的设备类型

首期支持：Cisco、Huawei、H3C 系列交换机/路由器。

### 5.6 可维护性

- 新增命令：编辑 `config/commands.yaml`，不改代码
- 新增设备类型：在配置中新增分类，并补充设备类型映射
- 凭据变更：修改 `.env` 文件

## 六、验收标准

### 6.1 功能验收

- [ ] AI 助手能调用「查看命令菜单」获取可用命令列表
- [ ] AI 助手能根据命令菜单选择 command_id，对单台设备执行查询并返回原始结果
- [ ] AI 助手能基于原始输出进行分析（如筛选 UP 接口、对比路由表）
- [ ] 批量查询多台设备能并发执行，返回每台设备各自的结果
- [ ] 客户端可通过传入 username/password 覆盖默认凭据
- [ ] 多命令单连接：一次调用传入多个 command_id，一次 SSH 连接内执行，结果按命令逐条返回
- [ ] 多命令 + 批量设备：多台设备各自一次连接执行所有命令，设备间并发
- [ ] 所有错误场景（连接失败、超时、认证错误等）均返回明确的错误原因，不返回空结果

### 6.2 安全验收（重点）

- [ ] 传入不存在的 command_id → 被拒绝
- [ ] 参数值包含注入字符（如 `; delete flash:`）→ 被拒绝
- [ ] 参数类型不匹配（如 ip_address 传入非法值）→ 被拒绝
- [ ] 必填参数缺失 → 被拒绝
- [ ] 拼装后命令如果触发兜底黑名单 → 被拒绝
- [ ] 空 command_id → 被拒绝
- [ ] 被拦截的操作会记录到审计日志

### 6.3 异常场景验收

- [ ] 设备不可达 → 返回连接失败信息并说明原因（如超时、拒绝连接、认证失败等），不影响其他设备
- [ ] 命令超时 → 返回超时提示，包含超时时长
- [ ] 批量执行中部分设备失败 → 成功的正常返回，失败的返回错误信息
- [ ] 批量超过 50 台 → 分批执行，不报错
- [ ] .env 缺少凭据且客户端未传入 → 返回明确的配置缺失错误
- [ ] 设备不支持某条内置命令（返回设备侧报错）→ 原样返回设备的错误输出，标记 success 但保留原文供 AI 判断
- [ ] SSH 认证失败（用户名密码错误）→ 返回认证失败信息
- [ ] 设备端口不通 → 返回连接被拒绝/超时的具体原因
- [ ] 多命令执行中某条命令出错 → 该命令标记失败，其余命令继续执行不受影响

### 6.4 日志验收

- [ ] 每次命令执行（成功或失败）都有日志记录
- [ ] 被安全拦截的操作也有日志记录，含拦截原因
- [ ] 日志文件位于 `logs/audit.log`，格式为每行一条 JSON
