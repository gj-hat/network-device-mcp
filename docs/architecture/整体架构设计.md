## 一、架构总览

### 1.1 系统定位

基于 MCP 协议的网络设备只读查询服务，**部署在远程服务器上**，通过 SSE（Server-Sent Events）传输与客户端通信。AI 助手通过 MCP 工具选择预定义命令并填写参数，服务端负责安全校验、命令拼装、SSH 执行和审计日志。

### 1.2 架构图（文字描述）

```
                                              ┌──── 远程服务器 ────────────────┐
                                              │                               │
用户 ──自然语言──▶ AI 助手        MCP (SSE)    │  ┌───────────────┐            │
                 （Claude Code）──────────────▶│  │  工具路由层    │            │
                                              │  └──────┬────────┘            │
                                              │         ▼                     │
                                              │  ┌───────────────┐            │
                                              │  │ 命令解析层    │            │
                                              │  └──────┬────────┘            │
                                              │         ▼                     │
                                              │  ┌───────────────┐            │
                                              │  │  安全校验层   │            │
                                              │  └──────┬────────┘            │
                                              │         ▼                     │
                                              │  ┌───────────────┐            │
                                              │  │ 凭据管理层    │            │
                                              │  └──────┬────────┘            │
                                              │         ▼                     │
                                              │  ┌───────────────┐   SSH     │
                                              │  │ SSH 执行层    │──────▶ 网络设备
                                              │  └──────┬────────┘           │
                                              │         ▼                     │
                                              │  ┌───────────────┐            │
                                              │  │  审计日志层   │            │
                                              │  └───────────────┘            │
                                              └───────────────────────────────┘
```

### 1.3 核心数据流

```
AI 传入: { command_id: "ping_test", params: { target: "10.0.0.1" }, host: "192.168.1.1", device_type: "cisco" }
    │
    ▼ 命令解析层
从 YAML 查到: command_id="ping_test" → 主命令="ping", 参数定义=[{name: target, type: ip_address}]
    │
    ▼ 安全校验层
校验参数值: "10.0.0.1" 符合 ip_address 类型 ✓
拼装最终命令: "ping 10.0.0.1"
兜底校验: 最终命令不含危险关键词 ✓
    │
    ▼ SSH 执行层
连接 192.168.1.1:22 → 发送 "ping 10.0.0.1" → 收取输出
    │
    ▼ 返回原始文本给 AI
```

## 二、模块设计

### 2.1 模块划分

| 模块 | 职责 | 关键依赖 |
|------|------|----------|
| `mcp_tools` | MCP 工具定义与路由，4 个工具的入口 | mcp sdk |
| `command_registry` | 加载 YAML 命令配置，提供命令查询、参数校验、命令拼装 | PyYAML |
| `security` | 参数值注入防护 + 拼装后命令兜底校验 | 无 |
| `credential` | 从 .env 读取默认凭据，支持客户端覆盖 | python-dotenv |
| `ssh_executor` | SSH 连接管理、单命令/多命令执行、超时控制 | netmiko |
| `audit_log` | 审计日志记录 | 标准库 logging |
| `config` | 全局配置（超时、并发上限等） | 无 |

### 2.2 模块间调用关系

```
mcp_tools
    ├──▶ command_registry   （查命令、校验参数、拼装命令）
    ├──▶ security           （兜底校验拼装后的命令）
    ├──▶ credential          （获取凭据）
    ├──▶ ssh_executor        （执行命令）
    └──▶ audit_log           （记录日志）
```

`mcp_tools` 是唯一编排层，其他模块之间不互相调用。

## 三、MCP 工具接口设计

### 3.1 工具一：单设备查询

**工具名**：`execute_readonly_command`

**输入参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| host | string | 是 | 设备 IP |
| device_type | string | 否 | cisco / huawei / h3c，默认 cisco |
| command_id | string | 是 | 命令标识（从命令列表获取） |
| params | object | 否 | 命令参数，key-value 形式 |
| port | integer | 否 | SSH 端口，默认 22 |
| username | string | 否 | SSH 用户名，覆盖服务端默认凭据 |
| password | string | 否 | SSH 密码，覆盖服务端默认凭据 |

**输出**：

```json
{
  "host": "192.168.1.1",
  "success": true,
  "command_executed": "ping 10.0.0.1",
  "output": "命令原始输出文本",
  "error": ""
}
```

### 3.2 工具二：批量查询

**工具名**：`batch_execute_readonly_command`

**输入参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| hosts | string[] | 是 | 设备 IP 列表 |
| device_type | string | 否 | 默认 cisco |
| command_id | string | 是 | 命令标识 |
| params | object | 否 | 命令参数 |
| port | integer | 否 | 默认 22 |
| username | string | 否 | SSH 用户名，覆盖服务端默认凭据 |
| password | string | 否 | SSH 密码，覆盖服务端默认凭据 |

**输出**：每台设备结果列表，格式同工具一。

**执行策略**：
- 命令解析与校验只做一次
- 各设备并发执行，并发上限 50（通过 asyncio.Semaphore 控制）
- 单台失败不影响其他设备

### 3.3 工具三：查看命令菜单

**工具名**：`list_available_commands`

**输入参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| device_type | string | 否 | 默认 cisco |

**输出**：命令列表，包含 command_id、名称、描述、参数定义。AI 根据此列表选择命令。

```json
[
  {
    "command_id": "show_version",
    "name": "查看设备版本",
    "description": "显示设备软件版本、硬件型号、运行时间等信息",
    "params": []
  },
  {
    "command_id": "ping_test",
    "name": "Ping 测试",
    "description": "从设备发起 ping 测试",
    "params": [
      { "name": "target", "type": "ip_address", "required": true, "description": "目标 IP 地址" }
    ]
  }
]
```

### 3.4 工具四：多命令单连接执行

**工具名**：`execute_multi_commands`

**场景**：巡检等需要一次获取多项信息的场景，一次 SSH 连接执行多条命令，减少重复建连开销。

**输入参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| host | string | 与 hosts 二选一 | 单台设备 IP |
| hosts | string[] | 与 host 二选一 | 多台设备 IP 列表 |
| device_type | string | 否 | 默认 cisco |
| commands | array | 是 | 命令列表，每项含 command_id 和可选 params |
| port | integer | 否 | 默认 22 |
| username | string | 否 | SSH 用户名，覆盖服务端默认凭据 |
| password | string | 否 | SSH 密码，覆盖服务端默认凭据 |

**commands 格式**：

```json
[
  { "command_id": "show_version", "params": {} },
  { "command_id": "show_interfaces_brief", "params": {} },
  { "command_id": "ping_test", "params": { "target": "10.0.0.1" } }
]
```

**输出（单台设备）**：

```json
{
  "host": "192.168.1.1",
  "results": [
    {
      "command_id": "show_version",
      "command_executed": "show version",
      "success": true,
      "output": "...",
      "error": ""
    },
    {
      "command_id": "show_interfaces_brief",
      "command_executed": "show ip interface brief",
      "success": true,
      "output": "...",
      "error": ""
    }
  ]
}
```

**输出（多台设备）**：每台设备一个上述结构的结果对象，组成列表返回。

**执行策略**：

- 每条命令**独立**进行安全校验，某条校验失败不影响其他命令
- 校验未通过的命令在结果中标记 `success: false`，`error` 说明拒绝原因
- 同一台设备：建立一次 SSH 连接，按顺序执行所有通过校验的命令
- 多台设备：各设备独立建连，设备间并发（受 Semaphore(50) 控制）
- 每台设备的连接超时仍为 30 秒

**与工具一、二的关系**：

| 工具 | 适用场景 | 连接策略 |
|------|---------|---------|
| 工具一 | 单设备单命令（简单查询） | 一次连接一条命令 |
| 工具二 | 多设备单命令（批量查询） | 每设备一次连接一条命令 |
| 工具四 | 单/多设备多命令（巡检） | 每设备一次连接多条命令 |

三个工具功能有重叠但定位不同，AI 根据场景复杂度选择最合适的工具。

## 四、命令配置设计

### 4.1 配置文件结构

配置文件路径：`config/commands.yaml`

命令列表参考来源：[ntc-templates](https://github.com/networktocode/ntc-templates) 项目，该项目按 `(platform, command)` 维护了主流网络设备的常用命令。Cisco 覆盖全面，Huawei / H3C 需按实际需求补充。

```yaml
# 按设备类型组织命令
cisco:
  - id: show_version
    name: "查看设备版本"
    description: "显示设备软件版本、硬件型号、运行时间等信息"
    command: "show version"

  - id: show_interfaces_brief
    name: "查看接口概要"
    description: "显示所有接口的 IP 地址和状态"
    command: "show ip interface brief"

  - id: show_interface_detail
    name: "查看指定接口详情"
    description: "显示指定接口的详细信息"
    command: "show interface"
    params:
      - name: interface_name
        type: string
        required: true
        description: "接口名称，如 GigabitEthernet0/1"
        pattern: "^[A-Za-z][A-Za-z0-9/.-]+$"

  - id: ping_test
    name: "Ping 测试"
    description: "从设备发起 ping 测试"
    command: "ping"
    params:
      - name: target
        type: ip_address
        required: true
        description: "目标 IP 地址"
      - name: count
        type: integer
        required: false
        description: "发包次数"
        range: [1, 10]
        default: 5

  - id: show_route
    name: "查看路由表"
    description: "显示设备完整路由表"
    command: "show ip route"

  - id: traceroute_test
    name: "路由跟踪"
    description: "从设备发起路由跟踪"
    command: "traceroute"
    params:
      - name: target
        type: ip_address
        required: true
        description: "目标 IP 地址"

huawei:
  - id: display_version
    name: "查看设备版本"
    description: "显示设备软件版本、硬件型号、运行时间等信息"
    command: "display version"

  # ... 华为命令列表

h3c:
  - id: display_version
    name: "查看设备版本"
    description: "显示设备软件版本、硬件型号、运行时间等信息"
    command: "display version"

  # ... H3C 命令列表
```

### 4.2 参数类型定义

| 类型 | 校验规则 |
|------|----------|
| `ip_address` | 合法 IPv4 地址 |
| `string` | 匹配 `pattern` 正则，必须提供 pattern |
| `integer` | 整数，可选 `range` 约束 |

**所有参数值均禁止包含 Shell 特殊字符**（`; | & \` $ \n \r \\`），这是通用规则，不依赖类型校验。

### 4.3 命令拼装规则

服务端拼装，AI 不参与：

```
最终命令 = 主命令 + " " + 参数值（按定义顺序拼接，空格分隔）
```

示例：
- `command: "ping"` + `params: {target: "10.0.0.1", count: 5}` → `"ping 10.0.0.1 count 5"`
- `command: "show version"` + 无参数 → `"show version"`

> **注意**：不同厂商的参数拼接格式可能不同（如 Cisco 的 `ping 10.0.0.1 repeat 5` vs Huawei 的 `ping -c 5 10.0.0.1`）。具体拼装格式在 YAML 中通过 `template` 字段扩展（详见后续迭代）。

## 五、安全设计

### 5.1 安全模型变化

相比 PRD 的三层开放式过滤，**命令配置化后安全模型本质改变**：

| | PRD 方案 | 配置化方案 |
|--|---------|-----------|
| 命令来源 | AI 自由输入 | 配置文件封闭集合 |
| 主防护 | 白名单前缀匹配 | command_id 查配置，不存在则拒绝 |
| 注入面 | 整条命令字符串 | 仅参数值（主命令不可变） |
| 黑名单 | 必需，作为兜底 | 保留作为防御纵深，但理论上不会触发 |

### 5.2 校验流程

```
AI 输入 command_id + params
        │
        ▼
   ① command_id 是否存在于当前 device_type 配置中？
        │ 否 → 拒绝: "未知命令"
        ▼ 是
   ② params 是否符合定义？（必填检查、类型校验、范围校验）
        │ 否 → 拒绝: "参数不合法"
        ▼ 是
   ③ 参数值是否包含注入字符？（; | & ` $ 等）
        │ 是 → 拒绝: "参数包含非法字符"
        ▼ 否
   ④ 拼装最终命令
        │
        ▼
   ⑤ 兜底：最终命令是否包含危险关键词？
        │ 是 → 拒绝（理论上不应触发，触发说明配置有问题）
        ▼ 否
   ⑥ 执行
```

### 5.3 凭据安全

**与 PRD 差异**：简化为一组默认凭据 + 客户端可覆盖，去掉多组凭据管理。

- 服务端 `.env` 中维护一组默认凭据：`SSH_USERNAME` / `SSH_PASSWORD`
- 客户端可通过 `username` + `password` 参数覆盖默认凭据
- 凭据优先级：客户端传入 > 服务端 `.env` 默认值
- `.env` 缺少默认凭据且客户端未传入时，返回明确错误
- 传输安全：MCP 使用 SSE 远程通信，凭据经过网络传输，**生产环境必须启用 HTTPS**

### 5.4 网络传输安全

当前阶段暂不做接入认证，客户端与服务端直接通信（HTTP + SSE）。

> **后续可增强**：HTTPS、API Key 认证、IP 白名单等，不在首期范围内。

## 六、SSH 执行层设计

### 6.1 技术选型

使用 **netmiko**（成熟的多厂商支持） + `asyncio.to_thread` 包装异步执行。首期追求稳定，不追求极致并发性能。

### 6.2 执行模式

SSH 执行层需支持两种模式：

**模式一：单命令执行**（工具一、工具二调用）

```
建立连接 → 发送 1 条命令 → 收取输出 → 断开连接
```

**模式二：多命令执行**（工具四调用）

```
建立连接 → 发送命令 1 → 收取输出 → 发送命令 2 → 收取输出 → ... → 断开连接
```

netmiko 的 `send_command()` 天然支持在同一个连接上多次调用，无需额外处理。多命令模式下，每条命令的输出独立返回，某条命令执行异常不中断连接，继续执行后续命令。

### 6.3 并发控制

- 使用 `asyncio.Semaphore(50)` 控制并发上限（以设备连接数为单位）
- 超出部分排队等待，不报错
- 单设备连接超时 30 秒

### 6.4 设备类型映射

netmiko 的 `device_type` 映射：

| 本系统 device_type | netmiko device_type |
|-------------------|---------------------|
| cisco | cisco_ios |
| huawei | huawei |
| h3c | hp_comware |

## 七、审计日志设计

### 7.1 日志原则

**每条命令一条日志**，不按连接或调用归并。即工具四一次执行 3 条命令，写 3 条日志。便于按命令维度检索和统计。

### 7.2 日志格式

每行一条 JSON，写入 `logs/audit.log`：

```json
{
  "timestamp": "2026-04-27T10:30:00+08:00",
  "tool": "execute_readonly_command",
  "host": "192.168.1.1",
  "port": 22,
  "device_type": "cisco",
  "credential_source": "default",
  "command_id": "ping_test",
  "command_executed": "ping 10.0.0.1",
  "params": {"target": "10.0.0.1"},
  "success": true,
  "error": "",
  "blocked": false,
  "block_reason": ""
}
```

`tool` 字段标识来源工具（`execute_readonly_command` / `batch_execute_readonly_command` / `execute_multi_commands`），便于区分调用场景。

### 7.3 记录时机

- 命令被安全校验拦截时：`blocked=true`，记录 `block_reason`
- 命令执行成功时：`success=true`
- 命令执行失败时（SSH 连接失败、超时等）：`success=false`，记录 `error`

## 八、扩展机制

### 8.1 新增命令

只需编辑 `config/commands.yaml`，在对应设备类型下新增条目，无需修改任何代码：

```yaml
cisco:
  - id: show_mac_table          # ← 新增
    name: "查看 MAC 地址表"
    description: "显示设备 MAC 地址表"
    command: "show mac address-table"
```

服务端启动时加载 YAML，AI 调用 `list_available_commands` 即可看到新命令。

### 8.2 新增设备类型

在 `config/commands.yaml` 中新增顶层 key，并在 SSH 执行层补充 netmiko `device_type` 映射即可。

### 8.3 MCP 工具与命令的关系

4 个 MCP 工具是固定的通用引擎，不与具体命令绑定：

```
┌─────────────────┐         ┌──────────────────┐
│  4 个 MCP 工具   │────────▶│  commands.yaml    │
│ （代码固定不变）  │         │ （唯一扩展点）     │
└─────────────────┘         └──────────────────┘
```

新增 100 条命令也不需要改一行代码。
